Prerequisite context

What This Work Is Actually Adding

The goal is to make a GrugMoE model trained and exported from Marin/Levanter run through the normal vLLM TPU serving path. vLLM remains the serving API and scheduler. tpu-inference owns the TPU/JAX model implementation and kernels. Marin owns the training model, export path, fork pins, and smoke tests that prove the pieces agree.

The PR is not just a registration change. GrugMoE has model semantics that existing supported transformer/MoE implementations do not already encode, so the TPU path needs custom model code, custom artifact loading, and custom diagnostics.

Quoted support QB-routed MoE with GatedNorm, XSA, sigmoid combine weights. Quoted support GrugMoE needs runner-owned JAX KV-cache and routed-expert plumbing.

1. Token setup

Start by defining the tiny toy values used by the rest of the walkthrough. These values are not checkpoint values; they are deliberately small stand-ins that make the Grug operations easy to inspect. 

import jax
import jax.numpy as jnp

# Toy shape knobs. The real Grug model uses much larger tensors.
K = 2  # Number of routed experts that run for each token.
eps = 1e-6  # RMSNorm denominator guard.

# One token hidden state with hidden_size = 4.
x = jnp.array([2.00, -1.00, 0.50, 1.50], dtype=jnp.float32)  # shape: [4]

# GatedNorm stand-ins. Real values come from checkpoint activations and weights.
gate_hidden = jnp.array([1.00], dtype=jnp.float32)  # Activation before the learned gate projection.
w_up = jnp.array([[2.20, 0.00, 0.85, 1.39]], dtype=jnp.float32)  # sigmoid(gate_hidden @ w_up) ~= [0.90, 0.50, 0.70, 0.80]

# Attention stand-ins used to show XSA and the head gate.
weighted_attn_out = jnp.array([0.80, 0.30], dtype=jnp.float32)  # Toy attention output for one head.
current_token_value = jnp.array([0.60, 0.80], dtype=jnp.float32)  # This token's own value vector.
attn_gate = jnp.array([-0.255, 0.0, 0.0, 0.0], dtype=jnp.float32)  # 2 * sigmoid(x @ attn_gate) ~= 0.75.

# Router and expert stand-ins.
router_logits = jnp.array([1.2, -0.4, 0.7, 0.1], dtype=jnp.float32)  # Unbiased router logits for E0..E3.
qb_bias = jnp.array([-0.3, 0.2, 0.0, 0.1], dtype=jnp.float32)  # Exported QB bias used only for expert choice.
expert_outputs = {
    0: jnp.array([0.20, -0.10, 0.40, 0.00], dtype=jnp.float32),  # Toy output from expert E0.
    2: jnp.array([-0.30, 0.50, 0.10, 0.20], dtype=jnp.float32),  # Toy output from expert E2.
}
shared_output = jnp.array([0.05, 0.02, -0.04, 0.01], dtype=jnp.float32)  # Toy shared SiLU MLP output.
  • This setup is the single source of toy values for all later states.
  • The real Grug block computes these tensors from checkpoint weights, but the operation order is the same.

Why Existing TPU Models Are Not Enough

Existing tpu-inference model classes cover standard dense transformers, ordinary RMSNorm, common MoE routing, and common checkpoint layouts. GrugMoE needs custom code because the implementation has to match all of the details above at once: QB-biased top-k selection, sigmoid mixing from unbiased logits, the boundary expert record, GatedNorm weights, exclusive self-attention post-processing, the shared dense MLP, and the Grug-specific safetensor names emitted by Marin.

That is why the work lands in all three places. Marin exports and tests the artifact. tpu-inference owns the JAX TPU implementation and its dense/reference plus production attention modes. vLLM only needs the narrow scheduler guard because the TPU runner can return routed expert IDs for diagnostics even when the user did not request routed-expert output.

How The Pieces Fit Together

Read the code as one pipeline. Levanter is the trusted training/reference implementation. Marin exports a canonical HF-style artifact. vLLM serves requests and schedules tokens. tpu-inference runs the JAX model on TPU and calls shared TPU attention kernels. Marin's tests compare the resulting logits, logprobs, generated tokens, and expert routes against the Levanter/reference path.

Dense Attention vs Production Attention

This distinction is central to the PR. The dense path is deliberately boring: compute attention directly over the visible tokens with explicit masks. It is slow and not a serving implementation, but it is easy to reason about and compare to Levanter. The production path uses the shared tpu-inference attention kernel, vLLM's request metadata, and the paged KV cache used during serving.

What should match? Hidden states, logits, selected logprobs, and routed experts under controlled prompts. Same outputs within tolerance, then successful installed vLLM canaries on real TPU.
What stays shared? QKV projection, rotary/q scaling, exclusive self-attention post-processing, headwise gate, MLP routing, shared expert, norms, and output logits.

The default exported artifact now says grugmoe_attention_mode = production, so an installed vLLM smoke exercises the real serving path. Dense mode remains available as the clean reference path.
