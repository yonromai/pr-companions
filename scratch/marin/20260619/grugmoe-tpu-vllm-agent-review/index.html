Prerequisite context

What This Work Is Actually Adding

The goal is to make a GrugMoE model trained and exported from Marin/Levanter run through the normal vLLM TPU serving path. vLLM remains the serving API and scheduler. tpu-inference owns the TPU/JAX model implementation and kernels. Marin owns the training model, export path, fork pins, and smoke tests that prove the pieces agree.

The PR is not just a registration change. GrugMoE has model semantics that existing supported transformer/MoE implementations do not already encode, so the TPU path needs custom model code, custom artifact loading, and custom diagnostics.

GrugMoE In One Pass

A normal transformer block has attention plus a feed-forward network. A normal MoE transformer replaces some feed-forward layers with a router that chooses a few experts. GrugMoE goes further: every block is an MoE block, the router uses Grug-specific QB routing, the block can add a shared dense expert, and the attention path uses Grug-specific normalization, gating, and exclusive self-attention.

Part What Reviewers Need To Know
Router Computes fp32 router logits, adds a non-gradient router bias for expert choice, selects top-(K+1), keeps top-K experts, and uses the extra expert as a QB boundary signal.
Combine weights Uses sigmoid of the unbiased selected router logits. This is not simply a softmax over the selected experts.
Experts Every layer has routed experts, and the canary shape exercises 64 experts with top-4 routing.
Shared expert A dense SiLU MLP is added beside the routed experts when shared_expert_intermediate_dim is positive.
Attention Uses q/k RMS norm, rotary embeddings, optional q scaling, grouped-query attention, short/long sliding windows, exclusive self-attention, and a headwise gate.
Norms GatedNorm appears around embeddings, attention input, MLP input, and final hidden states, so loader coverage must include those extra weights.

Why Existing TPU Models Are Not Enough

Existing tpu-inference models already know how to run standard dense attention, ordinary RMSNorm, common MoE routing, and common checkpoint layouts. GrugMoE differs in several places where changing one line in a registry would be wrong.

  • The router decision, combine weights, and returned expert IDs must match Levanter's Grug routing.
  • The attention output is post-processed by removing each token's own value direction, then applying a headwise gate.
  • The production serving path must use vLLM/tpu-inference KV cache metadata, while the debug path must stay simple enough to compare against Levanter.
  • The exported HF-style artifact has Grug-specific tensor names for GatedNorm, router bias, routed experts, and shared experts.
  • The vLLM scheduler can see routed expert data from the TPU runner, so it needs to ignore that data unless the routed-expert output feature is enabled.

That is why the work lands in all three places: Marin exports/tests, tpu-inference model/runtime code, and a narrow vLLM scheduler guard.

How The Pieces Fit Together

Read the code as one pipeline. Levanter is the trusted training/reference implementation. Marin exports a canonical HF-style artifact. vLLM serves requests and schedules tokens. tpu-inference runs the JAX model on TPU and calls shared TPU attention kernels. Marin's tests compare the resulting logits, logprobs, generated tokens, and expert routes against the Levanter/reference path.

Marin code

Defines the GrugMoE model, emits a canonical inference artifact, pins the fork-stack SHAs, and owns the smoke/parity harness.

tpu-inference fork

Adds the native JAX GrugMoeForCausalLM, loads the Marin artifact, executes the forward pass, updates KV cache, and returns routed expert IDs for diagnostics.

vLLM fork

Provides the serving front door and scheduler. The branch only needs a small guard so unsolicited routed expert data does not crash normal requests.

Dense Attention vs Production Attention

This distinction is central to the PR. The dense path is deliberately boring: compute attention directly over the visible tokens with explicit masks. It is slow and not a serving implementation, but it is easy to reason about and compare to Levanter. The production path uses the shared tpu-inference attention kernel, vLLM's request metadata, and the paged KV cache used during serving.

Question Dense/reference mode Production mode
Why exists? Debugging and Levanter parity. Real vLLM TPU serving.
Attention implementation Explicit QK scores, explicit mask, softmax, and value matmul. Shared tpu-inference attention with KV cache and full AttentionMetadata.
What should match? Hidden states, logits, selected logprobs, and routed experts under controlled prompts. Same outputs within tolerance, then successful installed vLLM canaries on real TPU.
What stays shared? QKV projection, rotary/q scaling, exclusive self-attention post-processing, headwise gate, MLP routing, shared expert, norms, and output logits.

The default exported artifact now says grugmoe_attention_mode = production, so an installed vLLM smoke exercises the real serving path. Dense mode remains available as the clean reference path.

Mental Model

Think of the dense path as the reference microscope and the production path as the real vLLM TPU serving route. For a small prompt, both paths receive the same parameters and token IDs. If the production path is correct, it should choose the same experts and produce nearly the same logits.

What To Scrutinize

Production attention correctness

Check that shared RPA gets full attention metadata, updates KV cache, applies the same exclusive self-attention post-processing as dense mode, and casts K/V to the cache dtype intentionally.

Debug mode remains simple

Dense attention should stay easy to read. It is a diagnostic/reference path, not the serving path to optimize.

Future TP/PP cost

The branch has explicit single-rank assumptions in the smoke harness and bespoke loading in the model implementation. That is acceptable for this diagnostic goal, but note the debt if pipeline or tensor parallel serving becomes urgent.
