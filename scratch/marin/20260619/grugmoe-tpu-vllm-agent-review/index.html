Prerequisite context

What This Work Is Actually Adding

The goal is to make a GrugMoE model trained and exported from Marin/Levanter run through the normal vLLM TPU serving path. vLLM remains the serving API and scheduler. tpu-inference owns the TPU/JAX model implementation and kernels. Marin owns the training model, export path, fork pins, and smoke tests that prove the pieces agree.

The PR is not just a registration change. GrugMoE has model semantics that existing supported transformer/MoE implementations do not already encode, so the TPU path needs custom model code, custom artifact loading, and custom diagnostics.

Normalization And Attention Terms

RMSNorm rescales a hidden vector by its root-mean-square magnitude, then applies a learned per-dimension weight. GrugMoE uses this before attention, before the MoE, after embeddings, and before final logits.

GatedNorm is an extra learned gate after RMSNorm. It projects the hidden vector down to a small rank-128 space, applies SiLU, projects back to hidden size, runs sigmoid, and multiplies the original normalized vector by that gate. It is not a standard vLLM normalization layer, so the loader and TPU model need Grug-specific weights and code.

Attention starts with normal Q/K/V projections, but normalizes Q and K, applies RoPE position rotation, optionally scales Q, uses grouped-query attention where several query heads share each K/V head, and alternates short and long sliding windows by layer.

Exclusive self-attention is the unusual part: after attention returns a head output y, Grug subtracts the part of y that points in the same direction as this token's own value vector v. The code computes the projection removal y - ((y · v) / ||v||2) v, then applies a learned scalar gate per attention head. Dense and production attention must both apply this same post-processing.

QB Routing And Expert Mixing

QB routing means Quantile Balancing. I found it sourced in the Marin notes to Jianlin Su's article-style quantile router-bias update, not to a formal paper. Operationally, each training step records a per-expert beta statistic from the router's top-k boundary. On the next step, Marin writes router_bias := -beta, mean-centered. That pushes over-used experts down and under-used experts up without using a normal auxiliary load-balancing gradient loss.

Non-gradient router bias means the bias is added for expert choice, but wrapped in stop_gradient. Backprop does not learn that bias directly. In an exported inference artifact, it is just a final tensor that the vLLM TPU path must load and add before top-k selection.

Boundary expert means the router asks for K + 1 experts. The first K are executed. The extra one is not executed; it gives the threshold logit used by QB statistics and by routing debug records.

Sigmoid combine weights are computed from the raw, unbiased logits of the selected experts. A selected-expert softmax would force the chosen expert weights to sum to one. Grug's sigmoid lets every selected expert independently contribute more or less, so the total contribution can be below or above one. The router bias affects which experts are chosen, not how strongly their outputs are mixed.

Shared dense SiLU MLP is an always-on feed-forward network that runs for every token beside the sparse routed experts. SiLU is the activation x * sigmoid(x). When shared_expert_intermediate_dim is positive, the final MLP output is routed-expert output plus this shared MLP output.

QB source Quantile Balancing (QB), a load balancing method for Mixture of Experts. Why Existing TPU Models Are Not Enough

Existing tpu-inference model classes cover standard dense transformers, ordinary RMSNorm, common MoE routing, and common checkpoint layouts. GrugMoE needs custom code because the implementation has to match all of the details above at once: QB-biased top-k selection, sigmoid mixing from unbiased logits, the boundary expert record, GatedNorm weights, exclusive self-attention post-processing, the shared dense MLP, and the Grug-specific safetensor names emitted by Marin.

That is why the work lands in all three places. Marin exports and tests the artifact. tpu-inference owns the JAX TPU implementation and its dense/reference plus production attention modes. vLLM only needs the narrow scheduler guard because the TPU runner can return routed expert IDs for diagnostics even when the user did not request routed-expert output.

How The Pieces Fit Together

Read the code as one pipeline. Levanter is the trusted training/reference implementation. Marin exports a canonical HF-style artifact. vLLM serves requests and schedules tokens. tpu-inference runs the JAX model on TPU and calls shared TPU attention kernels. Marin's tests compare the resulting logits, logprobs, generated tokens, and expert routes against the Levanter/reference path.

Dense Attention vs Production Attention

This distinction is central to the PR. The dense path is deliberately boring: compute attention directly over the visible tokens with explicit masks. It is slow and not a serving implementation, but it is easy to reason about and compare to Levanter. The production path uses the shared tpu-inference attention kernel, vLLM's request metadata, and the paged KV cache used during serving.
